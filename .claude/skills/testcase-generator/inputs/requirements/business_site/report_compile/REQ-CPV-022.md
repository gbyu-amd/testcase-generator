## 原始需求内容

### 【审批流程优化】- 再次提交自动上次带出审批人

【需求背景】

在计划管理、任务管理、方案管理、报告管理四个模块的审批流程中，用户经常需要多次修改后重新提交（例如被退回、驳回或撤回后修改）。当前每次重新提交都需要手动重新选择审批人，操作繁琐且容易遗漏或选错。为提高效率、减少重复工作，系统应支持在再次提交时自动带出上一次的审批人（包括原审批链、转办人、加签人），并允许用户编辑调整。复制创建的新记录不应带出审批人，避免误关联。

【需求描述】

|  |  |
| --- | --- |
| **项目** | **内容** |
| **需求描述** | **1. 适用模块范围**：计划管理、任务管理、方案管理、报告管理四个模块的提交审批操作。  **2. 触发场景**： - 审批人被退回后，提交人修改后再次提交。 - 审批人被驳回后，提交人修改后再次提交（与退回视为同义）。 - 提交人主动撤回审批单后，修改后再次提交。 - 审批过程中发生过转办或加签后，最终审批人拒绝或申请人撤回，再次提交时需带出最终的审批链（含转办人、加签人）。 - **注意**：首次提交时无历史审批人，按原有逻辑由用户选择或工作流自动确定。  **3. 带出规则**： - 系统自动查询该业务记录最近一次**完整的审批流程记录**（包括各级审批节点、每个节点提交时所选择的审批人、转办人、加签人）。 - 带出的审批人顺序与上次提交时的审批链顺序一致。 - 若上次审批过程中存在转办或加签，则带出提交时选择的人，再加上转办人和加签人 - 带出的审批人列表显示在提交页面的“审批人”字段中，用户可**编辑**（增加、删除、调整顺序）。  **4. 复制新建的处理**：通过“复制”功能创建的新记录（如复制一份计划），在提交时**不带出任何审批人**，视为全新记录。用户需手动选择或由工作流自动生成。  **5. 老数据兼容**：对于系统已有的历史数据（已完成审批、从未提交过等），在再次提交时若无法查询到上次审批记录，则按无历史记录处理，用户手动选择审批人。系统应能正确查询到已完成流程的审批链记录。 |
| **异常情况说明** | 1. 若查询上一次审批记录时出现异常或数据缺失，则提示“未能自动带出审批人，请手动选择”。 2. 带出的审批人中若有用户账号已被停用或已失效，则系统应提示“以下审批人已不可用：XXX”，并允许用户替换。 3. 若用户编辑后删除所有审批人，则无法提交，提示“请至少选择一位审批人”。 4. 转办/加签记录需完整存储，否则无法正确带出。需确保现有审计日志或流程历史表中包含这些信息。 |
| **后置处理功能** | 1. 用户再次提交后，系统以本次最终选定的审批人列表为准，覆盖原有的流程历史。 2. 记录审计日志：自动带出审批人的审计追踪同新增的审计追踪即可，若用户修改后提交则记录修改明细。 3. 对于复制新建的记录，在复制时明确清空审批人相关字段,审计追踪同之前； 4. 提供后台数据修复脚本，用于迁移老数据，确保历史记录的审批链可被正确查询。 |

**审批人带出规则补充表**

|  |  |  |
| --- | --- | --- |
| **场景** | **带出内容** | **可编辑性** |
| 上次提交后未发生转办/加签 | 带出原审批顺序列表（各级审批人） | 可增删改 |
| 上次提交后发生过转办（Transfer） | 带出转办后的实际审批人（原审批人不再出现） | 可增删改 |
| 上次提交后发生过加签（Add Sign） | 带出原审批人 + 加签人（按实际处理顺序） | 可增删改 |
| 多次转办/加签 | 带出最终生效的审批链（去重、按时间排序） | 可增删改 |
| 复制创建的新记录 | 不带出任何审批人 | 需手动添加 |
| 老数据（无完整流程记录） | 不带出，用户手动选择 | — |
