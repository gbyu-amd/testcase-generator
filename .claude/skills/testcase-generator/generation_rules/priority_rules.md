# 优先级规则

## 优先级取值

测试用例优先级只能使用：

- `P0`：阻塞性，必须在发布前通过，一旦失败直接阻断核心交易链路
- `P1`：重要，应在版本内完成，失败会显著影响用户体验或功能完整性
- `P2`：一般，有时间时覆盖，失败不影响核心交易但存在体验或边界问题

---

## P0 规则

满足以下**任一**条件，标记为 `P0`：

- 登录、购物车、结算、提交订单等**核心交易链路**的正向主流程。
- 订单金额、库存、优惠、地址等**关键数据一致性**验证。
- 可能导致**资损、重复订单、错误扣库存**的场景。
- **登录态过期**后仍可执行敏感操作（加购、提交订单）。
- 已知的**高风险操作路径**（重复提交、登录态失效后操作等）的回归验证。
- 接口失败导致页面无提示或**长时间卡死**的场景。
- 未登录用户访问购物车、结算、订单等**需要鉴权页面**。

**P0 典型示例：**

| 模块 | 典型 P0 场景 |
|---|---|
| 登录 | 正确账号密码登录成功；登录态过期后提交订单被拦截 |
| 购物车 | 失效商品不可参与结算；修改数量后金额同步更新 |
| 结算下单 | 正常提交订单成功；重复点击提交不生成重复订单 |
| 商品详情 | 已下架商品不可加购；切换规格后价格库存同步刷新 |

---

## P1 规则

满足以下**任一**条件，标记为 `P1`：

- 重要**分支流程**（如验证码登录、使用优惠券结算）。
- 常见**异常提示**场景（密码错误、库存不足、地址不可配送）。
- 筛选、排序、规格选择等**高频功能**。
- 商品失效、库存不足、价格变化等**重要状态变化**。
- 非核心但**影响用户购买决策**的功能（评价展示、收藏、加购成功提示）。
- 已知的**中等风险操作路径**（常见异常、重要状态变化）的回归验证。
- 弱网、接口超时等**接口异常**的页面表现。

**P1 典型示例：**

| 模块 | 典型 P1 场景 |
|---|---|
| 登录 | 验证码过期后提示重新获取；账号被冻结时禁止登录 |
| 搜索 | 筛选与排序条件叠加后结果正确；搜索无结果显示空状态 |
| 购物车 | 批量删除商品后列表和金额同步；购物车为空时展示空状态 |
| 结算下单 | 地址不可配送时阻断提交；优惠券过期时重新计算金额 |

---

## P2 规则

满足以下**任一**条件，标记为 `P2`：

- **低频边界场景**（超长名称截断、极大金额展示、价格为零）。
- **文案、样式、布局类**问题（字号、颜色、间距、对齐）。
- 搜索历史、辅助提示、收藏列表等**非主链路辅助功能**。
- iOS / Android 兼容性、横屏适配等**纯兼容场景**。
- 不影响核心交易链路的**体验优化**（加载动画、骨架屏）。
- 低风险操作路径或体验改善类场景的验证。

**P2 典型示例：**

| 模块 | 典型 P2 场景 |
|---|---|
| 搜索 | 搜索历史最多展示 10 条；关键词超长时截断显示 |
| 商品详情 | 商品名称超 50 字时省略展示；在 iPad 横屏下列表布局 |
| 我的足迹 | 足迹超 1000 条时滚动性能；价格为 0 时展示格式 |

---

## 优先级判断原则

无法直接判断时，按以下顺序评估：

1. **业务影响范围**：是否影响交易完成或产生错误数据？→ P0
2. **用户触达频率**：是高频操作路径，还是低频边界？→ 高频偏 P1，低频偏 P2
3. **是否有补救机制**：失败后用户是否能自助恢复？无补救 → 偏高优先级
4. **风险程度**：高风险路径 → P0，中等风险 → P1，低风险体验类 → P2

## 优先级分布建议

一个模块的完整用例集，优先级分布参考如下：

| 优先级 | 建议占比 | 说明 |
|---|---|---|
| P0 | 20%–35% | 聚焦核心链路，不应过多 |
| P1 | 40%–55% | 覆盖重要分支和常见异常 |
| P2 | 15%–30% | 边界、兼容、体验类补充 |

若 P0 用例超过 50%，说明优先级标注过于保守，应重新评估。
