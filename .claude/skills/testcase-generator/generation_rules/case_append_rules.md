# 用例补充生成规则

## 补充 vs 新建：先判断输出文件是否存在

| 情况 | 处理方式 |
|---|---|
| `outputs/origin_exports/<module_name>_testcases.md` 不存在 | 新建文件，编号从 001 开始 |
| 文件已存在 | 读取已有编号，按类型追加递增，不覆盖已有用例 |

---

## 补充生成流程

1. 判断新需求所属功能模块，确认模块名和输出文件路径。
2. 读取 `testcase_templates/modules/<module>/module_overview.md`（只读参考，不修改）。
3. 读取 `testcase_templates/modules/<module>/testcases.md`（只读参考，不修改）。
4. 若输出文件已存在，读取其中所有已有用例的编号、测试场景、前置条件和预期结果，用于去重判断。
5. 按去重规则（见下节）逐一判断新场景是否与已有用例重复。
6. 非重复的新场景，按编号递增规则（见 `testcase_id_rules.md`）在输出文件中追加。
7. 重复场景不新增用例，在本次输出中注明"已有覆盖，参见 <用例编号>"。
8. 追加完成后运行 `validate_cases.py`，确认整个输出文件无 ERROR。
9. 校验通过后运行 `export_testcases.py` 导出完整的 Excel（含已有用例和本次追加用例）。

---

## 编号追加规则

- 追加用例的编号必须在**同模块同类型**的现有最大序号基础上递增。
- 例如，文件内已有 `登录-功能-001` 至 `登录-功能-013`，本次追加的正向用例从 `登录-功能-014` 开始。
- 不同类型的编号序列**互相独立**：`登录-功能-014` 和 `登录-异常-001` 可以并存。
- 追加前必须扫描已有文件，不能凭记忆或猜测现有最大编号。

## 去重规则

### 判断流程

```
新场景 → 检查测试场景文字是否与已有用例完全相同？
  ↓ 是 → 重复，不新增
  ↓ 否 → 检查前置条件是否相同？
            ↓ 是 → 检查操作步骤是否高度相似（核心步骤一致）？
                      ↓ 是 → 检查预期结果是否相同？
                                ↓ 是 → 完全重复，不新增
                                ↓ 否 → 可能是场景变体，判断差异是否值得独立用例（见下方标准）
                      ↓ 否 → 不同操作路径，可以新增
            ↓ 否 → 不同前置数据，可以新增
```

### 差异是否值得独立用例的判断标准

以下差异**值得**独立用例（应新增）：
- 优先级不同（核心路径 vs 边界路径）
- 预期结果有实质差异（不同错误提示 / 不同跳转目标 / 不同数据状态）
- 前置条件涉及不同的账号角色或商品状态

以下差异**不值得**独立用例（应合并或忽略）：
- 仅步骤措辞不同，操作目标完全一致
- 数据值不同但验证目标相同（如密码 5 位 vs 密码 4 位，都是验证长度不足拦截）
- 与已有用例仅是顺序调整

### 重复时的处理

- 完全重复：不新增，注明"已有用例 <编号> 覆盖相同场景"。
- 部分重复但预期不同：在备注中说明差异，可新增独立用例并注明与 <编号> 的区别。
- 边界变体：若已有用例是最小值边界，新场景是最大值边界，则可以独立新增。

## 新模块处理

如果新需求不属于现有任何模块：

1. 不修改 `testcase_templates/modules/`（只读目录）。
2. 参考 `testcase_templates/common_templates/` 下的通用模板。
3. 基于用户提供的资料、`knowledge_base/` 和通用覆盖维度生成用例。
4. 保存到 `outputs/origin_exports/<new_module_name>_testcases.md`，编号从 001 开始。
5. 在输出文件开头标注：
   - 该模块是扩展模块
   - 缺失的参考资料（如无 PRD、无接口文档）
   - 生成假设（如"假设未登录用户访问该页面会跳转登录页"）

## 来源记录

每条新增用例必须记录来源，来源与 `testcase_writing_guidelines.md` 的来源字段定义一致：

| 来源值 | 使用场景 |
|---|---|
| 需求文档 | 来自本次输入的 PRD 功能需求或验收标准 |
| 界面设计图 | 来自 UI 截图或交互说明 |
| 接口文档 | 来自 `inputs/api_docs/` 中的接口定义 |
| 覆盖维度规则 | 来自 `coverage_dimension_rules.md` 中的回归场景要求 |
| 业务规则 | 来自 `knowledge_base/common_business_rules.md` |
| 覆盖维度规则 | 来自 `coverage_dimension_rules.md` 中的维度要求 |
| 用户角色 | 来自 `knowledge_base/user_roles.md` |

每条来源必须填写可溯源的值，不允许填写"通用"或"综合"。
