# 用例编写规范

## 基本原则

- 用例必须面向真实业务场景，不能脱离淘宝购物链路的上下文。
- 用例步骤必须可以被测试人员直接执行，不能依赖测试人员自行猜测前置数据。
- 预期结果必须可以观察或验证，包含页面表现、数据状态或业务状态中的至少一项。
- 不允许使用"正常""正确""可用"等模糊表达作为唯一预期。
- 核心交易链路必须覆盖成功、失败、回滚和重复操作。

---

## 字段规范

### 用例编号

- 遵守 `testcase_id_rules.md` 规定的格式：`模块名-类型-三位序号`。
- 同一输出文件内不允许重复。
- 编号中的模块名必须与"功能模块"字段完全一致。

### 功能模块

- 使用中文模块名，不加"模块"二字后缀。
- 合法值：`登录`、`搜索`、`商品详情`、`购物车`、`结算下单`、`首页`、`我的足迹`（及其他扩展模块名）。
- 同一用例只归属一个模块，跨模块场景以主操作模块命名。

### 测试场景

- 用一句话说明本条用例的验证目标，20 字以内为佳。
- 格式建议：`[操作对象] + [动作] + [关键条件]`，例如：
  - ✅ `登录态过期后点击结算按钮`
  - ✅ `购物车商品下架后勾选结算`
  - ✅ `搜索结果为空时的页面展示`
  - ❌ `测试登录功能`（过于宽泛）
  - ❌ `验证购物车`（无法定位具体场景）
- 同一模块内测试场景不允许重复。

### 优先级

- 只能使用 `P0`、`P1`、`P2`，遵守 `priority_rules.md` 的判断规则。

### 前置条件

- 必须写清所有外部依赖，包括：
  - 账号状态（已登录 / 未登录 / 账号冻结等）
  - 商品状态（正常在售 / 已下架 / 库存数量）
  - 地址（是否已设置默认收货地址、地址是否可配送）
  - 优惠（优惠券是否绑定、是否在有效期内）
  - 网络状态（正常 / 弱网 / 断网，异常场景必填）
  - 数据量要求（如"购物车内有 3 条以上记录"）
- 前置条件应能被测试人员独立复现，不能写"数据正常"。

**示例：**

| 写法 | 评级 |
|---|---|
| 用户已登录，购物车内有商品A（正常在售，库存 10 件），已设置默认收货地址 | ✅ 合格 |
| 数据正常 | ❌ 不合格 |
| 用户已登录 | ❌ 不完整（缺商品状态、地址等依赖） |

### 操作步骤

- 使用有序编号（`1.`、`2.`、`3.`），每步一行，步骤间用换行分隔。
- 每步描述一个独立操作，不合并多个动作。
- 至少 2 步，单步用例通常说明场景设计不完整。
- 步骤应包含操作对象和动作，例如：
  - ✅ `1. 打开淘宝首页`
  - ✅ `2. 点击顶部搜索框，输入关键词"手机"`
  - ✅ `3. 点击搜索按钮`
  - ❌ `1. 搜索商品`（动作与对象不明确）

### 预期结果

- 必须包含页面表现、数据状态或业务状态中的至少一项，不能少于 10 个字。
- 不允许使用以下空泛表达（单独作为唯一预期时）：

| 禁止写法 | 原因 |
|---|---|
| 页面展示正常 | 无可验证标准 |
| 功能可用 | 无法操作验证 |
| 结果正确 | 无法判断正确标准 |
| 按需求展示 | 依赖人工理解 |
| 展示成功 / 操作成功 / 通过 | 过于笼统 |

- 推荐写法：明确说明页面元素、数量、跳转目标、提示文案或数据变化。

**推荐示例：**

| 场景 | 推荐写法 |
|---|---|
| 正向登录 | 登录成功，页面跳转至首页，顶部展示用户昵称和购物车角标 |
| 库存不足提交订单 | 提交按钮置灰，页面提示"库存不足，请修改购买数量"，不生成订单 |
| 搜索无结果 | 页面展示"没有找到相关商品"空状态，提供"修改关键词"入口 |
| 删除足迹失败 | 提示"删除失败，请重试"，足迹记录保留在列表中，列表总数不变 |

### 用例类型

- 只能使用：`正向`、`异常`、`边界`、`权限`、`回归`、`兼容`、`联动`。
- 必须与用例编号中的类型字段对应（`功能` ↔ `正向`，其余一致）。
- 每条用例只能归属一种类型，若场景跨类型，以主要验证目的为准。

### 来源

- 必须填写，说明本条用例来自哪类输入，合法值：

| 来源值 | 适用场景 |
|---|---|
| 需求文档 | 来自 PRD 功能需求或验收标准 |
| 界面设计图 | 来自 UI 交互说明或界面元素 |
| 接口文档 | 来自接口状态码、错误码或幂等规则 |
| 覆盖维度规则 | 来自 `coverage_dimension_rules.md` 中的回归场景要求 |
| 业务规则 | 来自 `knowledge_base/common_business_rules.md` |
| 覆盖维度规则 | 来自 `coverage_dimension_rules.md` 中的维度要求 |
| 用户角色 | 来自 `knowledge_base/user_roles.md` 中的角色测试关注点 |

## 完整用例示例

以下为一条合格用例：

| 字段 | 内容 |
|---|---|
| 用例编号 | 结算下单-功能-001 |
| 功能模块 | 结算下单 |
| 测试场景 | 正常商品选中后提交订单成功 |
| 优先级 | P0 |
| 前置条件 | 用户已登录，购物车内有商品A（正常在售，库存 5 件，单价 99 元），已设置默认收货地址且可配送，无优惠券 |
| 操作步骤 | 1. 进入购物车页面<br>2. 勾选商品A<br>3. 点击"结算"按钮<br>4. 确认收货地址和订单金额<br>5. 点击"提交订单" |
| 预期结果 | 订单提交成功，页面跳转至支付页，展示订单编号、商品名称、数量和应付金额 99 元；购物车中商品A数量减少 1 |
| 用例类型 | 正向 |
| 来源 | 需求文档 |
