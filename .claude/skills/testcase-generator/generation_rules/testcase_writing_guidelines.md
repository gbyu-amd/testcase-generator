# 用例编写规范

## 基本原则

- 用例必须面向真实 CPV 业务场景，不能脱离制药企业持续工艺确认、站点隔离、审批合规和数据分析链路的上下文。
- 用例步骤必须可以被测试人员直接执行，不能依赖测试人员自行猜测前置数据。
- 预期结果必须可以观察或验证，包含页面表现、数据状态或业务状态中的至少一项。
- 不允许使用"正常""正确""可用"等模糊表达作为唯一预期。
- CPV 核心链路必须覆盖成功、失败、状态不允许、权限拦截、数据一致性、审计追踪和重复操作。
- 涉及审批、批准、生效、报告发布等关键动作时，必须验证电子签名、审批意见和审计记录。
- 涉及计划、任务、方案、监控项目、分析结果和报告之间的数据引用时，必须验证上下游状态和版本一致。

## 字段规范

### 分组字段

- `一级分组`、`二级分组`、`三级分组` 对应用例在 CPV 系统中的菜单或业务层级。
- `一级分组` 必须填写，优先使用 `公共管理站点` 或 `业务站点`。
- `二级分组` 填写一级菜单或业务域，例如 `公共管理`、`年度计划与任务`、`数据分析`。
- `三级分组` 填写二级菜单、页面或子流程，例如 `站点管理`、`年度计划设置`、`监控项目分析`。
- 同一输出文件内，同一业务模块的分组命名必须保持一致。

### 用例名称

- 用一句话说明本条用例的验证目标，20 字以内为佳。
- 格式建议：`[操作对象] + [动作] + [关键条件]`，例如：
  - 合格：`年度计划生效后生成周期任务`
  - 合格：`方案未生效时进入数据分析`
  - 合格：`替换数据源后字段删减`
  - 不合格：`测试计划功能`（过于宽泛）
  - 不合格：`验证数据分析`（无法定位具体场景）
- 同一分组内用例名称不允许重复。

### 优先级

- 只能使用 `P0`、`P1`、`P2`，遵守 `priority_rules.md` 的判断规则。

### 前置条件

- 必须写清所有外部依赖，包括：
  - 账号与角色（工艺工程师 / QA / QC / 生产操作员 / 系统管理员，账号启用或禁用）
  - 站点权限（公共管理 / 自定义业务站点，目标站点启用或禁用）
  - 菜单与按钮权限（是否有新增、提交、审核、批准、导出、分析等权限）
  - 业务数据状态（计划、任务、方案、模板、监控项目、报告的状态与版本）
  - 数据源与文件状态（数据表启用 / 失效，Excel 文件格式、字段、数据类型、范围）
  - 审批与电子签名配置（工作流节点、审批人、签名密码策略）
  - 网络状态（正常 / 弱网 / 断网，异常场景必填）
  - 数据量要求（如"监控项目下已有 3 个已分析配置项"）
- 前置条件应能被测试人员独立复现，不能写"数据正常"。

**示例：**

| 写法 | 评级 |
|---|---|
| 用户以工艺工程师账号登录自定义站点 A，拥有数据分析菜单和开始分析按钮权限，监控项目 M 处于未分析状态，待上传 Excel 文件包含批号、温度、压力字段 | 合格 |
| 数据正常 | 不合格 |
| 用户已登录 | 不完整（缺角色、站点、权限和业务数据状态） |

### 创建人

- 必须填写创建人姓名或团队约定账号。
- 新生成用例固定填写 `AI`，不随用户、模块或输入资料变化。

### 用例描述

- 用一句短语描述用例属性，推荐值：`正例`、`反例`、`边界`、`权限`、`兼容`、`回归`、`联动`。
- 该字段用于快速筛选，不替代用例名称和预期结果。

### 用例步骤

- 使用有序编号（`1.`、`2.`、`3.`），每步一行，步骤间用换行分隔。
- 每步描述一个独立操作，不合并多个动作。
- 至少 2 步，单步用例通常说明场景设计不完整。
- 步骤应包含操作对象和动作，例如：
  - 合格：`1. 登录自定义业务站点 A`
  - 合格：`2. 进入数据分析列表，点击监控项目 M 的"数据分析"按钮`
  - 合格：`3. 上传符合模板要求的 Excel 文件并点击"开始分析"`
  - 不合格：`1. 分析数据`（动作与对象不明确）

### 预期结果

- 必须包含页面表现、数据状态或业务状态中的至少一项，不能少于 10 个字。
- 不允许使用以下空泛表达（单独作为唯一预期时）：

| 禁止写法 | 原因 |
|---|---|
| 页面展示正常 | 无可验证标准 |
| 功能可用 | 无法操作验证 |
| 结果正确 | 无法判断正确标准 |
| 按需求展示 | 依赖人工理解 |
| 展示成功 / 操作成功 / 通过 | 过于笼统 |

- 推荐写法：明确说明页面元素、数量、跳转目标、提示文案或数据变化。
- 涉及 CPV 核心链路时，推荐同时说明上下游状态变化、引用关系、审计追踪或电子签名记录。

**推荐示例：**

| 场景 | 推荐写法 |
|---|---|
| 正向登录站点 | 登录成功，页面跳转至自定义站点首页，顶部展示当前站点名称和当前用户信息 |
| 年度计划生效 | 年度计划状态更新为已生效，任务管理中按计划周期生成对应任务，审计追踪记录批准人和生效时间 |
| 方案未生效进入分析 | 页面提示"方案未生效，暂不支持数据分析"，不生成监控项目分析记录 |
| 数据源字段删减 | 对应分析项显示"未分析"标签，悬停提示字段有删减，配置区域清空失效字段且不显示 `NULL` |
| 报告生效 | 报告状态更新为已生效，关联任务状态推进为完成，报告内容锁定并生成电子签名记录 |

### 备注、标签和关联字段

- 新生成或追加的用例，`备注` 必须填写来源，不允许为空或填写 `无`。
- 来自需求文档、PRD、验收标准或用户补充需求的用例，填写 `来源：需求文档`。
- 来自 UI 设计图、截图、原型图或界面交互说明的用例，填写 `来源：UI设计图`。
- 来自核心流程、业务规则、覆盖维度规则、优先级规则或具体知识库文件的用例，必须写清具体来源，例如 `来源：data_analysis_flow.md`、`来源：coverage_dimension_rules.md-合规追溯`、`来源：priority_rules.md-P0`。
- 同一条用例参考多个来源时，用分号分隔，例如 `来源：需求文档；来源：UI设计图；来源：common_business_rules.md`。
- 生成假设或评审说明不写入 `备注`，可写入文件元信息块或需求覆盖率对照表。
- `用例标签` 用于筛选，可填写 `P0`、`核心流程`、`权限`、`审计追踪`、`数据分析` 等标签，多个标签用分号分隔。
- `是否自动化`、`关联接口`、`用例测试类`、`关联项目` 保留字段，但新生成或追加用例的字段值必须留空。
