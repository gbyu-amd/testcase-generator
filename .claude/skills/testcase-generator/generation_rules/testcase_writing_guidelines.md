# 用例编写规范

## 基本原则

- 用例必须面向真实 CPV 业务场景，不能脱离制药企业持续工艺确认、站点隔离、审批合规和数据分析链路的上下文。
- 用例步骤必须可以被测试人员直接执行，不能依赖测试人员自行猜测前置数据。
- 预期结果必须可以观察或验证，包含页面表现、数据状态或业务状态中的至少一项。
- 不允许使用"正常""正确""可用"等模糊表达作为唯一预期。
- CPV 核心链路必须覆盖成功、失败、状态不允许、权限拦截、数据一致性、审计追踪和重复操作。
- 涉及审批、批准、生效、报告发布等关键动作时，必须验证电子签名、审批意见和审计记录。
- 涉及计划、任务、方案、监控项目、分析结果和报告之间的数据引用时，必须验证上下游状态和版本一致。

---

## 字段规范

### 标准输出字段

输出 Markdown 表格必须使用以下 16 列，顺序不可变：

| 一级分组 | 二级分组 | 三级分组 | 用例名称 | 优先级 | 创建人 | 用例描述 | 前置条件 | 用例步骤 | 预期结果 | 备注 | 用例标签 | 是否自动化 | 关联接口 | 用例测试类 | 关联项目 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|---|---|---|

### 分组字段

- `一级分组`、`二级分组`、`三级分组` 对应用例在 CPV 系统中的菜单或业务层级。
- `一级分组` 必须填写，层级简单时填写站点分类（`公共管理站点` 或 `业务站点`）；层级较多时可填写一级菜单或业务域（如 `审计追踪`、`报告编制`），站点信息体现在输出文件路径中即可。
- `二级分组` 填写下一层菜单或功能模块，例如 `角色管理`、`年度计划`、`数据分析`。
- `三级分组` 填写二级菜单、页面、操作类型或子流程，例如 `站点管理`、`年度计划设置`、`新增`、`批量删除`。
- 同一输出文件内，同一业务模块的分组命名必须保持一致。
- 同级分组下，用例必须按优先级排序：`P0` 排最前面，`P1` 排在 `P0` 后面，`P2` 排最后；相同优先级内保持业务流程或生成顺序。

### 用例名称

- 用一句话说明本条用例的验证目标，20 字以内为佳。
- 格式建议：`[操作对象] + [动作] + [关键条件]`，例如：
  - 合格：`年度计划生效后生成周期任务`
  - 合格：`方案未生效时进入数据分析`
  - 合格：`替换数据源后字段删减`
  - 不合格：`测试计划功能`（过于宽泛）
  - 不合格：`验证数据分析`（无法定位具体场景）
- 同一输出文件的同一分组内用例名称不允许重复；不同需求拆分到不同 Markdown 文件时，允许复用通用场景名称。

### 优先级

- 只能使用 `P0`、`P1`、`P2`，遵守 `priority_rules.md` 的判断规则。

### 前置条件

- 必须写清所有外部依赖，包括：
  - 用户类型与账号状态（系统管理员 / 普通用户，账号启用或禁用）
  - 站点权限（公共管理站点 / 自定义业务站点，目标站点启用或禁用）
  - 菜单、按钮和产品数据权限（业务站点角色是否启用，公共管理站点权限是否启用，是否有新增、提交、审核、批准、导出、分析等按钮权限）
  - 业务数据状态（计划、任务、方案、模板、监控项目、报告的状态与版本）
  - 数据源与文件状态（数据表启用 / 失效，Excel 文件格式、字段、数据类型、范围）
  - 审批与电子签名配置（工作流节点、审批人、签名密码策略）
  - 网络状态（正常 / 弱网 / 断网，异常场景必填）
  - 数据量要求（如"监控项目下已有 3 个已分析配置项"）
- 前置条件应能被测试人员独立复现，不能写"数据正常"。
- 前置条件只写执行用例前已经具备的环境、权限、数据和初始业务状态，不写后续操作或修复后的结果。
- 生成或评审用例时必须检查 `前置条件`、`用例步骤`、`预期结果` 是否重复表达同一件事：前置条件已写明的初始状态，步骤中不再用独立步骤复述；步骤中执行的动作，预期结果中不再重复描述操作过程。

**示例：**

| 写法 | 评级 |
|---|---|
| 普通用户登录业务站点 A，所属启用角色拥有数据分析菜单、开始分析按钮和产品数据权限，监控项目 M 处于未分析状态，待上传 Excel 文件包含批号、温度、压力字段 | 合格 |
| 数据正常 | 不合格 |
| 用户已登录 | 不完整（缺角色、站点、权限和业务数据状态） |

### 创建人

- 必须填写创建人姓名或团队约定账号。
- 新生成用例固定填写 `AI Agent`，不随用户、模块或输入资料变化。

### 用例描述

- 用一句短语描述用例属性，推荐值：`正例`、`反例`、`边界`、`权限`、`UI`、`兼容`、`回归`、`联动`。
- 来自 UI 设计图、截图、原型图或界面交互说明的页面区域整体展示校验，优先使用 `UI`；`兼容` 仅用于浏览器、分辨率、弱网、多页签等环境适配或稳定性场景。
- 该字段用于快速筛选，不替代用例名称和预期结果。

### 用例步骤

- 使用有序编号（`1.`、`2.`、`3.`），每步一行，步骤间用换行分隔。
- 每步描述一个独立操作，不合并多个动作。
- 至少 2 步，单步用例通常说明场景设计不完整。
- 步骤只写用户动作、系统触发动作或必要的页面确认，不写纯场景陈述；例如 `存在因样本不足处于未分析状态的箱线图` 应放在前置条件，步骤中写 `打开目标监控项目的数据分析详情页，确认箱线图为未分析状态且提示没有足够的数据`。
- 步骤应包含操作对象和动作，例如：
  - 合格：`1. 登录自定义业务站点 A`
  - 合格：`2. 进入数据分析列表，点击监控项目 M 的"数据分析"按钮`
  - 合格：`3. 上传符合模板要求的 Excel 文件并点击"开始分析"`
  - 不合格：`1. 分析数据`（动作与对象不明确）

### 预期结果

- 必须包含页面表现、数据状态或业务状态中的至少一项，不能少于 10 个字。
- 不允许使用以下空泛表达（单独作为唯一预期时）：

| 禁止写法 | 原因 |
|---|---|
| 页面展示正常 | 无可验证标准 |
| 功能可用 | 无法操作验证 |
| 结果正确 | 无法判断正确标准 |
| 按需求展示 | 依赖人工理解 |
| 展示成功 / 操作成功 / 通过 | 过于笼统 |

- 推荐写法：明确说明页面元素、数量、跳转目标、提示文案或数据变化。
- 涉及 CPV 核心链路时，推荐同时说明上下游状态变化、引用关系、审计追踪或电子签名记录。
- 来自 UI 设计图的校验应优先按页面区域整体描述，例如配置区 UI 校验、结果区 UI 校验。整体 UI 用例的预期结果必须列出覆盖的关键元素，不允许只写"页面展示正确"。
- 不应把静态说明文字、单个按钮、单个标题等拆成大量局部 UI 校验用例；只有该 UI 元素存在独立交互、业务规则、异常提示或边界限制时，才单独生成用例。

**推荐示例：**

| 场景 | 推荐写法 |
|---|---|
| 正向登录站点 | 登录成功，页面跳转至自定义站点首页，顶部展示当前站点名称和当前用户信息 |
| 年度计划生效 | 年度计划状态更新为已生效，任务管理中按计划周期生成对应任务，审计追踪记录批准人和生效时间 |
| 方案未生效进入分析 | 页面提示"方案未生效，暂不支持数据分析"，不生成监控项目分析记录 |
| 数据源字段删减 | 对应分析项显示"未分析"标签，悬停提示字段有删减，配置区域清空失效字段且不显示 `NULL` |
| 报告生效 | 报告状态更新为已生效，关联任务状态推进为完成，报告内容锁定并生成电子签名记录 |

### 备注、标签和关联字段

- 新生成或追加的用例，`备注` 必须填写来源，不允许为空或填写 `无`。
- 来自需求文档、PRD、验收标准或用户补充需求的用例，统一填写 `来源：prd`。
- 来自 UI 设计图、截图、原型图或界面交互说明的用例，填写 `来源：UI图`。
- 来自参考用例模板的用例，必须写清具体模板文件名，格式为 `来源：xxx.md`，例如 `来源：monitoring_items_paired_t.md`、`来源：report_generation.md`。
- 来自核心流程文件的用例，必须写清具体 `.md` 文件名，例如 `来源：data_analysis_flow.md`、`来源：one_click_analysis_flow.md`、`来源：cpv_business_flow.md`，不得写成中文“xxx核心流程”。
- 来自业务规则、覆盖维度规则、优先级规则或具体知识库文件的用例，必须写清具体来源，例如 `来源：coverage_dimension_rules.md-合规追溯`、`来源：priority_rules.md-P0`。
- 同一条用例参考多个来源时，用顿号分隔，例如 `来源：prd、UI图、monitoring_items_paired_t.md、data_analysis_flow.md`。
- 生成假设或评审说明不写入 `备注`，可写入文件元信息块或需求覆盖率对照表。
- `用例标签` 只填写按 `difficulty_level_rules.md` 判定的难度等级，取值只能为 `简单`、`一般`、`困难`。
- 不再写入业务筛选标签、优先级标签或模块标签，避免与当前标准表头和校验口径冲突。
- `是否自动化`、`关联接口`、`用例测试类`、`关联项目` 保留字段，但新生成或追加用例的字段值必须留空。
