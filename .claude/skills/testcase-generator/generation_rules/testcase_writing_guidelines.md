# 用例编写规范

## 基本原则

- 用例必须面向真实业务场景。
- 用例步骤必须可以被测试人员直接执行。
- 预期结果必须可以观察或验证。
- 不允许使用“正常”“正确”“可用”等模糊表达作为唯一预期。
- 核心交易链路必须覆盖成功、失败、回滚和重复操作。

## 字段规范

| 字段 | 编写要求 |
|---|---|
| 用例编号 | 遵守编号规则，保持唯一 |
| 功能模块 | 使用中文模块名 |
| 测试场景 | 一句话说明验证目标 |
| 优先级 | 只能使用 P0、P1、P2 |
| 前置条件 | 写清账号、商品、库存、地址、优惠等依赖 |
| 操作步骤 | 使用有序步骤，步骤之间用换行分隔 |
| 预期结果 | 包含页面表现、数据状态或业务状态 |
| 用例类型 | 正向、异常、边界、权限、回归、兼容、联动 |
| 来源 | 需求文档、界面设计图、接口文档、历史缺陷或业务规则 |

## 禁止写法

- 页面展示正常。
- 功能可用。
- 点击按钮后结果正确。
- 按需求展示。

## 推荐写法

- 页面展示商品名称、规格、单价、数量和库存状态。
- 提交订单按钮置灰，页面提示“库存不足”，系统不生成订单。
- 登录成功后跳转原访问页面，并展示用户昵称。
