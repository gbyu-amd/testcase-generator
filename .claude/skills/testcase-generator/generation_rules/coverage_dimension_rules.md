# 覆盖维度规则

## 使用原则

生成或补充 CPV 测试用例时，必须先按本文件评估覆盖范围，再结合 PRD、UI 设计图、知识库和参考用例生成具体场景。

- 每个功能模块至少评估“必须覆盖维度”中的 11 个维度。
- 与当前需求无关的维度，应在生成假设或需求覆盖率对照表中说明原因。
- 涉及 CPV 核心业务链路时，正向主流程、异常拦截、状态流转、数据一致性、合规追溯和跨模块联动必须重点覆盖。
- 新增或补充分析图 / 分析方法时，除 PRD 显式需求外，还必须检查“数据分析专项覆盖规则”。

---

## 必须覆盖维度

| 维度 | 覆盖重点 | 对应用例类型 |
|---|---|---|
| 正向主流程 | 用户按角色权限完成标准业务操作，如创建、提交、审批、生效、分析、报告输出 | 正例 |
| 异常流程 | 接口失败、网络中断、数据源失效、插件不可用、审批失败、异步任务失败等异常处理 | 反例 |
| 边界值 | 字段长度、文件大小、批次数量、周期范围、阶段数、控制限、规格限、日期区间等临界值 | 边界 |
| 字段与数据校验 | 必填项、格式、唯一性、数据类型、单位、范围、完整性、字段匹配和无效引用校验 | 反例 / 边界 |
| UI 设计图覆盖 | 页面区域、标题说明、字段、默认值、禁用态、按钮、表格列、结果区、提示文案和布局与 UI 设计图一致 | UI |
| 权限与站点隔离 | 未登录、登录态过期、无菜单权限、无按钮权限、无产品数据权限、无站点权限、跨站点访问拦截 | 权限 |
| 状态流转 | 草稿、审批中、已生效、退回、修订、失效、已分析、已确认、关闭等状态变化和可操作范围 | 正例 / 反例 |
| 数据一致性 | 计划、任务、方案、监控项目、相关性分析、分析结果、报告之间的引用、状态和版本一致 | 联动 / 回归 |
| 合规追溯 | 审计追踪、电子签名、审批意见、版本历史、操作前后值、失败原因和重试记录 | 回归 / 权限 |
| 跨模块联动 | 当前模块操作对下游任务、方案、监控项目、相关性分析、分析结果、报告、待办和通知的影响 | 联动 |
| 兼容性与稳定性 | 不同浏览器、分辨率、弱网、重复点击、页面刷新、返回、多页签、长耗时任务恢复 | 兼容 / 回归 |

| 模块类型 | 优先读取 |

## CPV 高风险场景

以下场景一旦涉及，必须生成对应用例，且最低优先级为 `P0`。

| 高风险场景 | 涉及模块 |
|---|---|
| 未生效年度计划生成正式周期性任务 | 年度计划、任务管理 |
| 失效模板被新方案或新报告引用 | 方案模板、方案编制、报告模板、报告编制 |
| 方案未生效即生成正式监控项目或进入分析 | 方案编制、监控项目、数据分析 |
| 未分析或未确认监控项目、相关性分析进入最终报告结论 | 监控项目、相关性分析、报告编制 |
| 报告生效后未推进关联任务状态 | 报告编制、任务管理 |
| 跨站点读取或写入计划、任务、方案、监控项目、相关性分析、报告数据 | 登录、站点管理、所有业务模块 |
| 无按钮权限或接口越权仍可提交、审批、批准、导出 | 权限管理、工作流、所有业务模块 |
| 无产品数据权限仍可查询、分析、导出或引用业务数据 | 业务站点角色管理、所有业务模块 |
| 审批、批准、生效未进行电子签名或未记录审计追踪 | 工作流、电子签名、审计追踪 |
| 已批准、已归档、已锁定数据被普通编辑覆盖 | 方案编制、监控项目、报告编制、基础数据 |
| 数据源失效、字段删减或 `NULL` 残留仍继续生成有效分析结果 | 数据分析、一键分析、监控项目、相关性分析 |
| 替换数据源后数据处理规则保留 / 清空逻辑错误 | 数据分析、一键分析 |
| 一键分析全部失败或部分失败时未提示未分析原因 | 数据分析、一键分析 |
| 分析结果回传失败但监控项目或相关性分析状态仍更新为已分析 | 数据分析、监控项目、相关性分析 |
| 报告引用跨任务、跨方案、跨站点或未确认的相关性分析结果 | 报告编制、监控项目、相关性分析、数据分析 |
| 批量导入、审批、导出、报告生成重复提交导致重复数据或重复任务 | 异步任务、年度计划、报告编制、数据分析 |
| 关键业务变更缺失操作人、时间、原因、前后值或版本历史 | 审计追踪、所有核心模块 |

## 覆盖检查表

生成用例后，按以下检查表逐项确认。

### 主流程与异常拦截

- [ ] 是否覆盖用户按角色权限完成当前模块核心操作的完整正向路径？
- [ ] 正向路径是否验证页面跳转、状态变化、业务数据落库和下游模块影响？
- [ ] 是否覆盖用户不能完成操作时的拦截提示，如无权限、状态不允许、数据无效、插件不可用、数据源失效？
- [ ] 拦截后是否验证系统没有执行被拦截的业务动作，如不生成任务、不生效方案、不更新监控项目或相关性分析状态、不推进报告状态？

### UI 设计图

- [ ] 是否已从 UI 设计图提取页面元素清单，包括页面区域、标题说明、字段、默认值、禁用态、按钮、表格列、结果区、提示文案和布局？
- [ ] UI 元素清单中的每个元素是否能追溯到具体用例，不允许只在脑内判断已覆盖？
- [ ] 是否优先按页面区域生成整体 UI 校验用例，例如配置区 UI 校验、结果区 UI 校验，而不是为每个静态文字或按钮拆成零散局部用例？
- [ ] 对有明确业务规则或交互行为的 UI 元素，是否已单独生成业务用例，例如重置、保存、开始分析、默认值、禁用态、必填校验、字数限制？
- [ ] `来源：UI图` 的用例是否集中覆盖 UI 结构和关键展示，避免大量重复、碎片化的局部校验用例？

### 状态、审批与合规

- [ ] 是否覆盖草稿、审批中、退回、已生效、修订、失效、关闭等状态下的可操作范围？
- [ ] 是否覆盖提交、审核、批准、退回、生效等工作流动作及电子签名要求？
- [ ] 是否覆盖审计追踪、电子签名、审批意见、操作前后值、版本历史和失败原因记录？
- [ ] 是否确认已批准、已归档、已锁定数据不会被普通编辑动作覆盖？

### 数据一致性与跨模块联动

- [ ] 是否覆盖计划、任务、方案、监控项目、相关性分析、分析结果、报告之间的引用一致性？
- [ ] 是否验证跨页面、跨模块、刷新、返回、重新进入详情页后的数据与服务端状态一致？
- [ ] 是否覆盖本模块操作对待办、通知、任务进度、监控项目状态、相关性分析状态、报告状态和异步任务的影响？
- [ ] 是否覆盖多页签、浏览器返回、页面刷新、弱网和重复点击场景？

### 站点与权限

- [ ] 是否覆盖公共管理站点和自定义业务站点的菜单边界、数据隔离和站点切换？
- [ ] 是否覆盖业务站点角色的菜单权限、按钮权限、产品数据权限、启停和多角色权限并集？
- [ ] 是否覆盖公共管理站点权限管理中的菜单权限、按钮权限、启停和多权限并集？
- [ ] 是否覆盖接口越权、登录态失效和用户禁用后的访问控制？

### 数据分析专项覆盖

**通用能力**
- [ ] 是否覆盖文件校验、插件状态、算法配置、字段配置、动态表单和开始分析前校验？
- [ ] 是否覆盖数据分析工作台、监控项目和相关性分析三类入口的分析图范围差异？
- [ ] 生成“分析方法列 / 选择算法”类用例时，是否已确认 PRD 中当前分析图所在的 `选择算法-<分类名>` 上级标题，并在用例名称、步骤、预期结果中使用该分类原文？例如 P控制图应使用 `质量控制图`，不得自行概括为 `控制图分类`。
- [ ] 生成变量区域、字段匹配、数据替换、数据处理和一键分析相关用例时，是否已逐项核对当前分析图 / 分析方法在 PRD 或 UI 设计图中的变量名称、变量数量和必填性？不同分析图的变量名称不得互相套用，例如 P控制图按当前 PRD/UI 使用 `纵轴变量`、`横轴变量`，不得从其他分析图或参考模板沿用 `图形变量`、`样本1`、`样本2`、`样本量字段` 等不属于当前配置区的字段名称。
- [ ] 生成分析结果页用例前，是否已按页面区域分别摘录字段和元素清单？至少区分 `分析流程`、`算法配置`、`分析图表`、`过程数据`、`判异检查`、`参考线检查`、`输入结论`，不得跨区域复用字段。
- [ ] 生成 `过程数据` 相关用例时，是否只引用过程数据表中实际出现的字段？`过程数据与 minitab 一致`、`过程数据字段显示正确`、`小数位数对过程数据生效` 等用例，均不得引用仅出现在分析图、算法配置或系统结论中的字段。
- [ ] 是否覆盖数据源预览、替换数据源、数据处理、一键分析、未分析原因优先级、字段删减清理和数据源失效？

**分析图 / 分析方法默认覆盖项**
新增或补充任一分析图 / 分析方法用例时，不论 PRD 是否显式描述，均需检查以下默认覆盖项：

| 默认覆盖项 | 检查要求 |
|---|---|
| 变量区域基础交互 | 删除变量后回到选择变量列表；点击添加定类变量失败；拖拽第二个定量变量失败；点击添加第二个定量变量失败 |
| 未分析首次进入字段匹配 | 数据源字段匹配不上则不带出，匹配上则带出 |
| 数据源预览 | 从监控项目进入数据分析详情页，数据源处预览数据，显示正确 |
| PNG/JPG 导出 | 导出 PNG/JPG 图片后，图片内容仅包含分析图表；分析图表内容与页面展示保持一致，不覆盖分析流程、详细结论、过程数据、判异检查、参考线检查或输入结论等结果页其他区域 |
| 分析结果名称 | 重命名分析结果名称后监控项目列表同步更新；已分析结果在分析流程顶部显示当前分析结果名称 |
| 分析图标题 | 分析图标题不以 UI 图中的示例数据为准，默认按 `<纵轴变量>_<分析图名称>` 生成；当前分析图没有纵轴变量时，按当前分析方法已确认的变量命名规则生成 |
| 小数位数默认选项 | 小数位数不存在为空或未选择状态，默认必须有选中值；生成用例时需校验默认值和下拉选项，但具体默认值、选项范围必须以当前分析方法的 PRD、UI 设计图或已确认业务规则为准 |
| 小数位数影响范围 | 小数位数会影响哪些结果区域或字段，必须以当前分析方法的 PRD、UI 设计图或已确认业务规则为准；资料未明确时，可参考模板保留该覆盖维度，但不得从参考模板或其他分析方法推断为影响过程数据、分析图、判异检查、参考线检查或系统结论；对应用例备注必须将来源和待确认说明分开，写作：来源：coverage_dimension_rules.md-小数位数影响范围；说明：需求文档未明确需要确认 |
| 不同数据类型 | 不同数据类型覆盖必须优先以当前分析方法的 PRD、UI 设计图或已确认业务规则为准；当前分析方法明确不支持的数据类型，应归入对应异常 / 边界流程处理，不再作为“不同数据类型”默认覆盖项重复生成；若资料未明确限制，则默认需要支持小数列、负数列、百分位数列、断层混合型数据列均能按当前分析方法规则处理 |

上述默认覆盖项应参考 `testcase_templates/modules/business_site/scheme_execution/monitoring_items_box_plot.md`、`testcase_templates/modules/business_site/scheme_execution/monitoring_items_paired_t.md` 中对应用例风格，并结合当前分析图 / 分析方法的字段名称、变量数量、结果结构和异常规则改写预期结果。

### 一键分析专项覆盖

新增或补充任一分析图 / 分析方法用例时，一键分析为默认强制覆盖项，不以 PRD 是否显式描述为前提。

- [ ] 是否已读取 `knowledge_base/core_flows/one_click_analysis_flow.md`？
- [ ] 是否已参考 `testcase_templates/modules/business_site/scheme_execution/monitoring_items_box_plot.md`、`testcase_templates/modules/business_site/scheme_execution/monitoring_items_paired_t.md` 中一键分析相关用例风格？
- [ ] 是否覆盖一键分析五类分组：`数据替换`、`数据处理`、`未分析的图参与一键分析`、`定类变量暂不支持分析`、`其他原因`？
- [ ] `未分析的图参与一键分析` 是否已结合当前分析图 PRD 的异常 / 边界流程处理生成，覆盖字段匹配、数据修复、样本量 / 数据量 / 变量值等条件恢复后重新参与一键分析，以及仍不满足条件时保持未分析并提示原因？
- [ ] 当前分析图 / 分析方法存在多个变量配置项时，是否已按 PRD 或 UI 设计图列出每个变量的缺失、字段不匹配和修复后参与一键分析场景？变量缺失类用例必须按变量拆分，不得只生成一条泛化用例；例如 P控制图需分别覆盖 `纵轴变量缺失` 和 `横轴变量缺失`，配对T检验需分别核对 `样本1`、`样本2` 等当前方法实际变量。
- [ ] `其他原因` 是否已结合当前分析图 PRD 的异常 / 边界流程处理生成，例如样本量不足、数据量不一致、变量值全相同、阶段数超限等当前分析图适用的异常？
- [ ] 同一异常若既适用于手动开始分析又适用于一键分析，是否已补充一键分析下的未分析状态、悬浮原因、配置保留 / 清空和修复后重新分析结果？

### 优先级分布

- [ ] P0 用例是否聚焦在 CPV 核心链路、合规风险、数据完整性、权限隔离和报告结论准确性？
- [ ] P0 占比是否在 20%–35% 合理区间？若超过 50%，应重新评估是否标注过高。
