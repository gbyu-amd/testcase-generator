# 覆盖维度规则

## 必须覆盖维度

每个功能模块生成用例时，至少评估以下 9 个维度，每个维度至少 1 条用例：

| 维度 | 说明 | 对应用例类型 |
|---|---|---|
| 正向主流程 | 用户完成核心操作的标准路径 | 正向 |
| 异常流程 | 接口失败、数据异常、操作被拦截 | 异常 |
| 边界值 | 数量、长度、金额、时间等临界值 | 边界 |
| 字段校验 | 必填项缺失、格式错误、超长输入 | 异常 |
| 权限控制 | 未登录、登录态过期、账号异常 | 权限 |
| 状态变化 | 商品下架、库存变化、优惠失效 | 异常 / 正向 |
| 回归场景 | 覆盖已知的高风险操作路径，防止同类问题复现 | 回归 |
| 跨模块联动 | 当前模块操作对其他模块的数据或状态影响 | 联动 |
| 兼容性风险 | iOS / Android、横屏、弱网、不同设备分辨率 | 兼容 |

## 各模块特有维度

### 登录模块（核心链路）

除必须覆盖维度外，还需覆盖：

| 额外维度 | 示例场景 |
|---|---|
| 登录态失效 | 长时间停留后 token 过期，继续操作被拦截并回跳 |
| 多页签状态同步 | 一个页签退出登录，另一个页签识别未登录状态 |
| 登录成功回跳 | 从购物车 / 结算 / 商品详情触发登录后，登录成功回到原页面 |
| 重复提交防护 | 点击登录按钮后按钮禁用，不重复发送请求 |
| 账号异常 | 冻结账号、风险账号、需要二次验证 |

### 搜索模块

| 额外维度 | 示例场景 |
|---|---|
| 空结果展示 | 搜索无匹配商品时展示空状态和引导 |
| 条件叠加正确性 | 筛选 + 排序同时生效，翻页后条件保持 |
| 分页连续性 | 上拉加载不重复、不遗漏商品 |
| 关键词边界 | 空关键词、超长关键词、特殊字符 |

### 商品详情模块

| 额外维度 | 示例场景 |
|---|---|
| 规格切换同步 | 切换 SKU 后价格、库存、主图同步刷新 |
| 商品失效状态 | 下架商品展示提示，加购 / 购买按钮置灰 |
| 库存边界 | 数量选择不超过库存上限，达上限时禁止增加 |
| 页面返回保持 | 从详情页返回列表页，滚动位置保持 |

### 购物车模块（核心链路）

除必须覆盖维度外，还需覆盖：

| 额外维度 | 示例场景 |
|---|---|
| 数据一致性 | 数量、单价、小计、合计金额实时同步 |
| 失效商品处理 | 下架商品不可勾选，不计入结算金额 |
| 重复操作 | 快速多次点击加减数量，结果应准确 |
| 多页签同步 | 一个页签修改购物车，另一个页签刷新后数据一致 |
| 接口失败回滚 | 修改数量失败后，页面还原到修改前数量 |

### 结算下单模块（核心链路）

除必须覆盖维度外，还需覆盖：

| 额外维度 | 示例场景 |
|---|---|
| 重复提交防护 | 连续点击提交订单按钮，不生成重复订单 |
| 数据一致性 | 商品金额、优惠金额、应付金额三者一致 |
| 失败回滚 | 提交失败后订单不生成，库存不扣减 |
| 外部数据变化 | 提交前商品下架 / 库存不足 / 价格变化，提交被拦截并提示 |
| 地址校验 | 地址为空 / 不可配送时禁止提交 |
| 弱网处理 | 弱网下按钮禁用避免重复提交，提示超时 |
| 页面刷新 / 返回 | 刷新或返回后库存、价格、优惠重新校验 |

### 首页模块

| 额外维度 | 示例场景 |
|---|---|
| 接口失败降级 | 轮播图 / 楼层接口失败时有占位，不影响其他区域 |
| 登录态联动 | 登录 / 退出后个性化内容刷新 |
| 快捷入口跳转 | 金刚区每个入口跳转到正确目标页 |

### 我的足迹（扩展模块）

| 额外维度 | 示例场景 |
|---|---|
| 分组边界 | 今天 / 昨天 / 更早分组时间临界点正确 |
| 批量操作状态 | 进入 / 退出编辑模式时选中状态重置 |
| 数据过期清理 | 90 天后足迹不展示 |

## 淘宝高风险维度（任何模块均应评估）

以下场景一旦涉及，必须生成对应用例：

| 高风险场景 | 涉及模块 | 最低优先级 |
|---|---|---|
| 商品下架后仍可购买 | 商品详情、购物车、结算下单 | P0 |
| 库存不足时仍可提交 | 商品详情、结算下单 | P0 |
| 价格变化后未重新计算 | 购物车、结算下单 | P0 |
| 优惠券失效后仍抵扣 | 结算下单 | P0 |
| 地址不可配送时仍可提交 | 结算下单 | P0 |
| 订单重复提交 | 结算下单 | P0 |
| 购物车失效商品参与结算 | 购物车、结算下单 | P0 |
| 账号被冻结仍可操作 | 登录 | P0 |
| 登录态失效后仍可执行敏感操作 | 登录、购物车、结算下单 | P0 |

## 覆盖检查表

生成用例后，逐项确认以下问题：

**主流程覆盖**
- [ ] 是否有用例覆盖用户能完成核心操作的完整正向路径？
- [ ] 正向路径是否验证了跳转目标、展示数据和状态变化？

**拦截覆盖**
- [ ] 是否有用例覆盖用户不能完成操作时的拦截提示？
- [ ] 拦截后是否验证了系统没有执行被拦截的操作（如不生成订单、不扣库存）？

**数据一致性**
- [ ] 是否有用例覆盖数据变化后页面是否同步刷新？
- [ ] 是否验证了多个模块/页面间展示的数据一致？

**回归场景**
- [ ] 是否覆盖了已知的高风险操作路径（如重复提交、登录态失效后操作）？
- [ ] 回归用例是否还原了该场景的完整触发条件？

**跨模块联动**
- [ ] 是否有用例覆盖本模块操作对相关模块的影响？
- [ ] 是否覆盖了多页签 / 浏览器返回 / 刷新场景？

**优先级分布**
- [ ] P0 用例是否聚焦在核心链路和高风险场景？
- [ ] P0 占比是否在 20%–35% 合理区间？
