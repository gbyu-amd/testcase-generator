# 用例编号规则

## 编号格式

用例编号采用以下格式：

```text
模块名-类型-三位序号
```

示例：

- `登录-功能-001`
- `搜索-边界-001`
- `商品详情-异常-001`
- `购物车-回归-001`
- `结算下单-功能-001`

## 类型取值

| 编号类型 | 对应用例类型字段 | 含义 |
|---|---|---|
| 功能 | 正向 | 正常业务流程 |
| 异常 | 异常 | 错误、失败、状态异常 |
| 边界 | 边界 | 数量、长度、金额等边界 |
| 权限 | 权限 | 登录态、账号状态、角色限制 |
| 回归 | 回归 | 历史缺陷修复验证 |
| 兼容 | 兼容 | 浏览器、设备、网络兼容 |
| 联动 | 联动 | 跨页面或跨模块状态同步 |

## 编号类型与用例类型字段的对应关系

注意：**编号中的"类型"**与**输出表格中的"用例类型"字段**取值大部分一致，唯一区别是：

- 正常业务流程在**编号**中写作 `功能`（例如 `登录-功能-001`）。
- 同一条用例的**用例类型字段**写作 `正向`，不是 `功能`。

其余六种类型（异常、边界、权限、回归、兼容、联动）在编号和用例类型字段中写法完全相同。

校验脚本会强制检查这一对应关系：若编号类型为 `功能` 而用例类型字段不是 `正向`，或两者不匹配，将报 `case_id_type_mismatch` 错误。

## 递增规则

- 同一输出文件内，同一模块、同一类型下按三位序号递增。
- 生成新输出文件时，从 `001` 开始编号。
- 追加到 `outputs/origin_exports/<module_name>_testcases.md` 时，必须读取该输出文件内已有编号并继续递增。
- `testcase_templates/` 中的参考用例编号只用于理解场景，不参与本次输出编号。

## 新模块规则

新增模块时，应先确定中文模块名和输出文件名，再从 `001` 开始编号并保存到 `outputs/origin_exports/`。
