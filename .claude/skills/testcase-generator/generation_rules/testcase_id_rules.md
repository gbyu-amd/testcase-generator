# 用例编号规则

## 编号格式

用例编号采用以下格式：

```text
模块名-类型-三位序号
```

**规则：**
- 模块名必须与"功能模块"字段完全一致（不加"模块"二字后缀），例如功能模块为"登录"，编号模块名也写"登录"，不写"登录模块"。
- 类型只能使用下方"类型取值"表中的合法值。
- 三位序号：001、002、…、999，不足三位补零。

**合法示例：**

- `登录-功能-001`
- `搜索-边界-001`
- `商品详情-异常-001`
- `购物车-回归-001`
- `结算下单-功能-001`
- `我的足迹-权限-002`

**非法示例（校验会报 ERROR）：**

- `登录模块-功能-001`（模块名多了"模块"后缀）
- `登录-功能-1`（序号不足三位）
- `login-001`（缺少类型段）
- `登录-正向-001`（类型字段写了"正向"而非"功能"）

---

## 类型取值

| 编号类型 | 对应用例类型字段 | 含义 |
|---|---|---|
| 功能 | 正向 | 正常业务流程 |
| 异常 | 异常 | 错误、失败、状态异常 |
| 边界 | 边界 | 数量、长度、金额等边界 |
| 权限 | 权限 | 登录态、账号状态、角色限制 |
| 回归 | 回归 | 高风险操作路径的回归验证 |
| 兼容 | 兼容 | 浏览器、设备、网络兼容 |
| 联动 | 联动 | 跨页面或跨模块状态同步 |

## 编号类型与用例类型字段的对应关系

注意：**编号中的"类型"**与**输出表格中的"用例类型"字段**取值大部分一致，唯一区别是：

- 正常业务流程在**编号**中写作 `功能`（例如 `登录-功能-001`）。
- 同一条用例的**用例类型字段**写作 `正向`，不是 `功能`。

其余六种类型（异常、边界、权限、回归、兼容、联动）在编号和用例类型字段中写法完全相同。

校验脚本会强制检查这一对应关系：若编号类型为 `功能` 而用例类型字段不是 `正向`，或两者不匹配，将报 `case_id_type_mismatch` 错误。

---

## 递增规则

- 同一类型的编号序列**互相独立**：`登录-功能-014` 和 `登录-异常-001` 可以并存，序号各自从 001 开始。
- 同一输出文件内，同模块同类型下按三位序号**连续递增**，不允许跳号（校验脚本会报 WARN）。
- 生成新输出文件时，每种类型均从 `001` 开始编号。
- 追加到已有输出文件时，必须读取该文件内同模块同类型的最大序号，在其基础上递增，不能凭记忆猜测。
- `testcase_templates/` 中的参考用例编号只用于理解场景，不参与本次输出编号。

**递增示例：**

假设 `outputs/origin_exports/login_testcases.md` 已有：

| 已有编号 | 类型 |
|---|---|
| 登录-功能-001 ~ 013 | 正向 |
| 登录-异常-001 ~ 015 | 异常 |
| 登录-边界-001 ~ 012 | 边界 |

本次追加时：
- 新正向用例从 `登录-功能-014` 开始
- 新异常用例从 `登录-异常-016` 开始
- 新边界用例从 `登录-边界-013` 开始
- 首次新增权限用例从 `登录-权限-001` 开始

---

## 新模块规则

1. 确定中文模块名（与功能模块字段一致）。
2. 确定输出文件名：`outputs/origin_exports/<module_name>_testcases.md`（使用英文或拼音文件名，不含空格）。
3. 所有类型均从 `001` 开始编号。
4. 不修改 `testcase_templates/modules/` 目录。
